# Phase 05 Plan 01: Resilience and Quota Management

## Objective

Add a Room-backed video ID cache so warm playlist runs cost near-zero search quota, and confirm that artists with no YouTube search result are gracefully skipped without aborting the generation run.

## Context

- ArtistEntity already exists in Room (artist_history table, schema v2)
- AppDatabase is at version 2; adding the video_id_cache table requires migration to v3
- YouTubeRepository.searchTopSong() returns null when no result found
- GeneratePlaylistUseCase already filters null results and counts them as skipped (YT-03 is implemented)
- The main work is YT-02: video ID cache in Room

## Tasks

### Task 1: Add VideoIdCacheEntity and VideoIdCacheDao (auto)

Create Room entity `VideoIdCacheEntity` and DAO `VideoIdCacheDao`:
- Entity table: `video_id_cache`
- Primary key: `normalizedArtistName` (String, artist name lowercased and trimmed)
- Fields: `videoId` (String, nullable — null means "no result found for this artist"), `cachedAt` (Long, epoch millis)
- DAO operations: `getByArtistName(name: String)`, `upsert(entity: VideoIdCacheEntity)`, `deleteAll()`

### Task 2: Add DB migration v2 -> v3 and wire DAO (auto)

- Add `MIGRATION_2_3` to `AppDatabase` that creates the `video_id_cache` table
- Increment database version to 3
- Add `videoIdCacheDao()` abstract function to AppDatabase
- Register migration in `DatabaseModule.provideDatabase()`
- Add `@Provides fun provideVideoIdCacheDao(db: AppDatabase)` to DatabaseModule

### Task 3: Wire cache into YouTubeRepository (auto)

Modify `YouTubeRepositoryImpl` and `YouTubeRepository` interface:
- Inject `VideoIdCacheDao` into `YouTubeRepositoryImpl`
- In `searchTopSong()`: check cache first (lookup by normalized artist name); if hit, return cached videoId; if miss, call API, then write result to cache (including null results — null means "known no-result")
- Cache hit on null videoId means skip the API call and return null immediately (quota saving for known no-result artists)

### Task 4: Update FakeYouTubeRepository and tests (auto)

- Update `GeneratePlaylistUseCaseTest` to verify:
  - When an artist has a cached video ID, searchTopSong is not called for that artist (verify via call count tracking on FakeYouTubeRepository)
  - Artists with null video IDs are skipped and counted correctly
- Add a simple unit test for VideoIdCacheDao using in-memory Room database

## Verification

- All existing tests pass
- New tests pass: VideoIdCacheDaoTest + GeneratePlaylistUseCaseTest additions
- App builds without errors

## Success Criteria

1. Room database has a `video_id_cache` table (schema v3) — warm runs use cache instead of YouTube API
2. Artists returning null from YouTube search are counted as skipped in the success summary — one missing artist does not abort the run (already implemented, verified by existing tests)
